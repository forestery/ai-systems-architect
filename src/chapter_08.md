## 8.1 这个角色到底做什么

AI Systems Architect 这个名字还没有行业标准定义。Gartner 提过，Forbes 提过，但没人写过职位描述。以下是基于 2026 年招聘市场和实际实践的综合画像。

### 核心职责

```
AI Systems Architect = Agent 编排架构的设计者
                     + Context Engineering 的所有者
                     + 质量/安全体系的构建者
                     + 人-Agent 协作流程的设计者
```

展开说：

- **设计 Agent 编排架构。** 不是写代码——是决定用什么范式（Orchestrator-Worker 还是 Planner-Worker-Judge）、怎么拆任务、怎么配置 Agent 集群。
- **构建和维护 Context Engineering 系统。** 项目知识怎么组织、语义检索怎么配置、Agent 记忆怎么持久化——这些是你的基础设施。它们不直接产生代码，但它们决定产出的代码对不对。
- **设计质量体系。** 不是跑测试——是设计「什么情况下合并、什么情况下拒绝、什么情况下升级给人」的规则。三层验证是你最重要的工具。
- **管理人-Agent 协作流程。** 人什么时候介入、Agent 什么时候自主执行、异常怎么路由——这些流程的设计比任何单个技术决策都重要。

### 一周长什么样

| 时间 | 活动 | 占比 |
|------|------|------|
| 上午 | 翻译意图——把业务需求拆成 Agent 可执行的任务分解 | 30% |
| 中午 | 管理上下文——更新项目 Context、调语义检索、审查 Agent Memory | 20% |
| 下午 | 审查和介入——看 Agent 产出报告、处理升级的异常、做合并决策 | 30% |
| 全天 | 自己被中断频率：低。Agent 只在你设定的升级条件下找你 | — |

对比传统 Tech Lead 的一天：被各种人打断、回答「这个怎么做」、处理人际冲突。AI Systems Architect 被中断的频率低得多——大多数问题在执行层就被 Agent 消化了，只有真正的异常才会上升到人的桌面。

## 8.2 你现在能做什么

自我评估是最好的起点。以下是一份能力清单——不是「未来需要的能力」，是**现在市场上已经在被定价的能力。**

### 能力自检

```
编排能力
  ☐ 能用单 Agent 完成一个独立功能的开发
  ☐ 能用 Orchestrator-Worker 模式编排 3+ Agent 协作
  ☐ 能设计适合自己项目的编排规则（非照搬模板）
  ☐ 能分析 Agent 失败的原因并调整编排策略

上下文工程
  ☐ 能为自己的项目写结构化的上下文（不只是 README）
  ☐ 能搭建基本的语义检索（向量库 + 嵌入模型）
  ☐ 能设计分层注入策略（Layer 1/2/3）
  ☐ 能维护 Agent Memory（跨会话学习）

质量控制
  ☐ 能配置自动验证流水线（lint + test + build）
  ☐ 能设计交叉验证流程
  ☐ 能定义人类审查的触发条件（而非每件事都审）
  ☐ 能识别 Agent 代码的典型犯错模式

安全意识
  ☐ 能在 Agent 上下文中加入安全约束
  ☐ 能识别 Agent 产出中的常见安全问题
  ☐ 能设计 Agent 工作空间的隔离策略
```

数一下你打了几个勾。0-3 个：你在第一阶段。4-7 个：第二阶段。8 个以上：你已经接近这个角色的市场要求。

## 8.3 正在发生的技能迁移

观察 2025-2026 年的工程招聘市场，一个趋势在逐渐清晰：**对「编码速度」的定价在下降，对「编排能力」的定价在上升。** 这不是突然的断裂——更像是重心在缓慢但确定地移动。

具体来说，三个变化值得注意：

**第一，团队效率的基准变了。** 传统上一个三人组分担的工作量，现在一个熟练使用 Agent 编排的工程师就可以覆盖。这并不意味着团队会立刻缩编——但它正在改变对单个工程师产出的期望值。

**第二，Agent 做不了的判断在变得更贵。** 决定什么不该构建。在模糊需求中抓出逻辑矛盾。预判一个架构决策在六个月后的瓶颈。这些判断力一直是高级工程师的核心价值——但过去它们和编码能力混合在一起。现在编码层被 Agent 覆盖之后，判断力这一层单独浮出水面，而且正在被单独定价。

**第三，Context Engineering 正在成为一个独立职位。** 一年前，「管理 Agent 的上下文」是 Prompt Engineering 的延伸。现在它正在变成一项独立的工程实践——有自己的工具栈、设计模式和考核指标。Anthropic 的 2026 年报告称之为「承重技能」，不是没有原因的。

## 8.4 技能市场在怎么变

基于 2026 年 Q2 的招聘市场观察：

| 级别 | 薪资范围 (USD) | 对应条件 |
|------|---------------|---------|
| 入门 | $150K-$180K | 3-5 年 SE + 能独立用 Agent 完成完整项目 |
| 中级 | $200K-$250K | 5-8 年 SE + 有投产的多 Agent 编排经验 |
| 高级 | $250K-$300K+ | 8+ 年 SE + 能设计企业级编排架构 |

> 说明：这个角色太新，样本量有限。以上数据来自多个招聘平台和行业报告的综合估算，不是精确统计。

对比传统 Senior Engineer 的中位数（美国，$180K-$220K），AI Systems Architect 在同一经验水平上有明显的溢价。中国市场的数据更难拿，但 2026 年 Q1，几家大厂在「AI 开发效能」方向的招聘预算明显高于传统后端/前端岗位。

Agent 做不了的事情清单在 2026 年还很长：它不知道不该构建什么、它不会拒绝模糊的需求、它看不出一个架构决策会在六个月后成为瓶颈、它不理解用户为什么愤怒。这些全是人类判断力的领域。

## 8.5 从哪里开始

这本书到这里就结束了。以下是这份地图提供的几条参考路径：

### 本周

1. 如果还没有，选一个 Agent 编码工具（Claude Code 或 Codex），装好
2. 用这个工具完成你今天的所有编码任务。不对比。直接让它做
3. 记录：Agent 做对了什么？做错了什么？你在什么情况下不信任它？

### 这个月

4. 为你的项目写第一版结构化上下文（参考第 5 章第 5.3 节）
5. 在三个真实任务中尝试 Orchestrator-Worker 编排（参考第 6 章模板）
6. 建立一个 Agent 失败日志——每次 Agent 犯错，写下根因（上下文不够？spec 歧义？模型能力边界？）

### 这三个月

7. 用 Agent 完成一个完整的、可以上线的项目——从 spec 到部署，每一步都是 Agent 做的，你只在关键决策点介入
8. 复盘：过去三个月，Agent 在什么地方反复出错？你的任务分解有什么模式性的缺陷？
9. 把你学到的东西讲给一个同事听。教的过程会暴露你理解不够深的地方

完整的 12 个月计划在[附录 F](appendix_f.md)里，供参考。

## 地图在这里

前七章覆盖了我目前关于多 Agent 编程的关键认知——架构、范式、角色、上下文工程、编排、质量。它不是完整的。这个领域在以月为单位变化。

但有一个方向越来越清晰：**软件工程的重心正在从「构建」迁移到「编排」。** 这不是预言——是正在发生的技术演进。迁移的赢家不是最早用 Agent 的人，是理解「怎么让 Agent 正确工作」的人。

> **本章留一句话：** 编程不只是写代码。验证也是编程。
