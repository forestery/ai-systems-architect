# 第 4 章 · 角色坍缩：谁留下，谁消失

---

2025 年底，一家硅谷 B 轮 SaaS 公司的 CTO 在内部做了一个决定，然后写了一篇博客解释他的理由。事情本身不大，但他写出来的那句话被转得到处都是：

> 「我们这个季度招了三个工程师。但我觉得其中两个其实可以不开。」

他说这话并不是在夸 AI 有多厉害。他在算一笔很具体的账。他们团队已经在用 Claude Code 做日常开发——已经到了每个工程师每天都用的程度。产出上去了。但他发现，产出上去的原因不是每个人都变快了，而是**原来需要三个人的活，现在一个会用 Agent 的人就能干完。**

「多出来的两个人」，他说，「不是能力不行。是他们的工作方式在被重新定价。」

这篇博客底下的评论吵了几百楼。有骂他冷血的。有说他说出了别人不敢说的。有说自己公司也在发生同样的事，只是没人写出来。

在这个 CTO 的账本上，「3 个人变 1 个」这个算式不是预测——是他已经看到的数字。

这就是第 4 章要讨论的事。不是「未来会怎样」，是「现在正在发生什么」。你属于哪个角色，你的角色再往后推三年会变成什么样，以及你可以做什么。

---

## 4.1 团队的形状正在变

Allstacks 是一家做工程效能分析的公司。他们在 2026 年初出了一份预测报告，里面有一张表让我盯了很久。

这是他们的「2030 年软件团队结构预测」：

```
2024 年典型团队 (8 人)              2030 年预测团队 (4 人 + Agent)
───────────────────                ──────────────────────
Engineering Manager    (1)         EM + AI Systems Architect  (1-2)
Tech Lead              (1)         
Senior Developer       (2)         Senior Dev / Orchestrator  (1)
Junior Developer       (2)         Agent 集群                 (N)
QA Engineer            (1)         QA Strategist              (0.5)
PM                     (1)         PM                         (1)
```

八个人变成了最多四个人，加上一批不占编制的 Agent。

这张表本身是一个咨询公司的预测，不是既成事实。但它指向的方向，和你听到的、看到的、感觉到的是同一个方向。它不是在预测「可能会发生」——它是在把已经在发生的趋势画到五年后的样子。

拆开来看每个角色。

---

## 4.2 初级开发者：不是消失，是入门路径改了

受影响最大的群体。原因很直接：Agent 最擅长的就是初级开发者的核心工作。

写 CRUD 接口。写单元测试。写样板代码。按已有 spec 实现功能。这些事情 Agent 做得更快、不出语法错误、不嫌无聊。

但「受影响最大」不等于「死路」。入门路径在变，不是门关上了。

### 旧的入门路径

```
学编程语言 → 写简单功能 → 积累编码经验 → 
  写更复杂的功能 → 开始参与设计 → 成为 Senior
```

这个过程的核心资源是「写代码的时间」。你写的行数越多，你越熟练。

### 新的入门路径

```
学编程语言 + 学 Agent 工具 → 给 Agent 写 spec → 
  审查 Agent 的输出 → 理解 Agent 为什么对、为什么错 →
  从审查中积累判断力 → 成为 Agent Orchestrator
```

这个过程的核心资源不再是「你写的行数」，而是**「你审查了多少行 Agent 写的代码，你从中学到了多少判断力」。**

坦白说，这对刚入行的人来说更难。旧路径的反馈更直接——写出来的代码能跑就是对的。新路径的反馈更间接——Agent 写的代码能跑可能还是错的，你需要比过去更早地建立「判断对错」的能力，而不仅仅是「把事情做出来」的能力。

但这里面有一个被低估的机会：**初级 Orchestrator 现在极其稀缺。** 市场上有一大把会用 Copilot 的初级开发，但能在 Claude Code 里编排两个子 Agent 协作完成一个完整功能的人——这种人很少。不是因为他们能力不够，是因为没人告诉他们「你应该往这个方向走」。

### 一个具体的建议

如果你入行不到三年，做两件事：

1. **停止比较你和 Agent 谁写代码更快。** 那个比较已经结束了。Agent 更快。
2. **开始练习「把一个大任务拆成三个小任务，交给两个 Agent 和一个你自己，然后判断结果」。** 这个能力——任务分解 + 质量判断——就是三年后你的市场定价基础。

## 4.3 资深开发：你的手艺没贬值，但定价方式变了

资深开发者的处境比初级开发者复杂。不是更差——是更需要想清楚自己在卖什么。

### 什么东西在快速贬值

「写得快」——Agent 更快。
「写得不出错」——Agent 在简单任务上几乎不出错。
「能搞定一个独立模块」——Agent 也行，而且不需要工时评估。

这些曾经是资深开发定价的核心。它们没有「消失」——你仍然可以自己写代码，仍然可以比初级开发者更快——但**市场不再为这些事情支付溢价了。** 因为 Agent 可以做到同样的事，边际成本趋近于零。

### 什么东西在升值

「知道该构建什么」——Agent 不知道。它只能执行 spec。
「知道为什么不构建某些东西」——Agent 不知道。它不会拒绝。
「能从 PRD 里抓出三处逻辑矛盾」——Agent 偶尔能做到，但不可靠。
「能设计一个系统，让 Agent 在其中高效工作」——这是全新技能，而且极度稀缺。

资深开发者的转型方向不是「放弃技术」——恰恰相反，是**把技术判断力往上移一层。** 你不再判断「这段代码写得好不好」——Agent 写的代码你审查就行。你判断的是「这个架构能不能让 Agent 在六个模块里同时工作而不会互相踩脚」。

### 新的身份

最自然的转型方向是 **Agent Orchestrator**——不是管人，是管 Agent 集群：

- 把需求翻译成 Agent 可执行的任务分解
- 管理 Agent 的工作空间和上下文
- 审查 Agent 的产出——不是一行行看代码，是判断设计决策和边界条件
- 当 Agent 卡住时，知道该换方案还是该自己上手

我知道有一个在 Zoom 工作了八年的 Senior Engineer，去年转了 Agent Orchestrator。他说了一段让我印象深刻的话：

> 「前三个月特别难受。我觉得自己什么都没做——一天下来一行代码都没写。但三个月后我发现，我做的事从『实现功能』变成了『保证功能被正确实现』。产出没变少，变的是产出不再刻着我的名字。」

## 4.4 Tech Lead：管人变成管 Agent，但判断力没变

Tech Lead 的处境有一个优势和一个劣势。

优势：这个角色本来就不以「写多少行代码」来定价。它的核心一直是判断力——技术选型、架构方向、风险识别、人的协调。这些东西 Agent 暂时做不了，可能长期也做不了。

劣势：TL 现在需要管的不只是人，还有 Agent。而且 Agent 的失败模式和人的失败模式完全不同。

### 新增的工作

| 传统 TL 的职责 | 多出来的 Agent 相关职责 |
|--------------|---------------------|
| 分配任务给人 | 分配任务给人 **和** Agent，知道什么该给谁 |
| 评估人的进度 | 评估 Agent 的吞吐——Agent 可能看起来很快但方向全错 |
| 解决人际冲突 | 解决 **Agent 之间的协调冲突**——两个 Worker 在不同的 worktree 里改了同一个文件，谁来合并？用什么策略？ |
| 知道谁在瓶颈上 | 知道 **Agent 的失败模式**——它是真的不会做，还是上下文不够，还是模型版本的问题 |

最后一个能力特别关键。当一个 Agent 反复失败，TL 需要快速判断：这个任务该换一个 Agent 模型来做？还是该重新分解？还是该自己上手？还是该告诉 PM「这个需求 Agent 做不了」？

这个判断力的好坏，直接决定团队的 Agent 利用率是 80% 还是 30%。

### 不变的东西

有一个最重要的东西没有变，也不会变：**TL 仍然要对所有技术决策负责。** 不管多少行代码是 Agent 写的，出 Bug 了不是你怪 Agent——是你错了。你设的编排方式让 Agent 出了错。或者你的审查流程漏了这个 Bug。

这个压力没有减轻。但这也正是 TL 这个角色不会被 Agent 取代的原因——**责任和判断力是同一枚硬币的两面，而 Agent 不承担任何一面的重量。**

## 4.5 PM 和 QA：两个人的价值都在涨

PM 和 QA 是两个不同的角色，但放在一起讲，因为 Agent 对他们的影响方向是一样的：**价值提升。**

### PM：从需求管理到意图定义

传统 PM 的核心工作之一是把模糊的业务需求翻译成开发能看懂的 spec。这个工作没有因为 Agent 消失——反而变得更关键了。

在人类开发的模式下，PM 写的 spec 有歧义，开发会来问。在 Agent 开发的模式下，**PM 写的 spec 有歧义，Agent 不会问——它会按自己的理解直接做。** 做出来的东西可能是错的，但因为你没有人类开发那个「这一看就不对」的常识判断，错误可能到测试阶段才发现，甚至到上线后。

所以 PM 的 spec 质量在 Agent 时代变成了**直接影响产出准确性的变量。** 好的 spec = 好的软件。坏的 spec = Agent 高速产出一堆坏软件。

我见过的最好的实践是：PM 写完 spec 后，先丢给一个 Agent（不是执行 Agent，是一个专门的「spec 审查 Agent」）检查逻辑一致性。Agent 标记出三个矛盾点。PM 修改后再发给执行 Agent。这个流程的成本几乎为零，收益是减少了一半的返工。

### QA：从手工测试到质量系统设计

传统 QA 做很多手工测试——点来点去、填表单、看结果。Agent 可以覆盖所有这些——而且更快、更勤快、不会漏。

但 QA 的价值不在「执行测试」，而在**决定测试什么。** 一个好的 QA 知道一个支付系统的边界条件：余额刚好等于支付金额、同时发生了网络超时、第三方回调抵达时本地数据库正在做快照——会发生什么？

Agent 不会想到这个场景。QA 会。QA 的新定位是从「测试执行者」变成 **「测试策略设计师」** ——定义验证标准，设计边界条件测试，审查 Agent 自动生成的测试是否真的覆盖了所有需要覆盖的东西。

## 4.6 推演到 2030 年

回头看 Allstacks 那张表。如果你觉得「3-4 人 + Agent」这个结论太激进，那换一个问法：**如果一个会用 Agent 的人能干以前三个人的活，一个 20 人的工程团队应该保留多少人？**

我见过三种回答：

1. 「20 人不变，产出翻三倍。」——不现实。公司的需求不会在两年内翻三倍。而且维护 3 倍的代码量需要更多的维护人力，可能反而回到原点。

2. 「裁到 7 人，节省成本。」——会有人这么干，但不会是真正的赢家。AI Systems Architect 极度稀缺，你裁掉一半的人可能正好裁掉了你最需要留住的人。

3. 「20 人重组为 12 人 + Agent 集群，产出不变但质量更高。」——这是我看到的走得最稳的方案。不是拿 Agent 替代人，是**让 Agent 接管重复层，让人集中到决策层。** 人的数量不会从 20 降到 4，但人的工作内容会发生根本性变化。

2030 年的团队不会是「几个人 + 一大堆 Agent」这种科幻画面。更可能的是一张混合的、正在演化中的组织结构图——上面有一些传统角色不动（PM、核心架构师），有一些角色转到了新的定义（Agent Orchestrator、QA Strategist），还有一些角色——说实话——被 Agent 消化了。

问题不是你在这张图上的哪个格子里。问题是：**你正在主动走向你想要的格子，还是等着被移到某个你不想去的格子里。**

> **本章只有一个观点：** 软件工程的各种角色正在被重新分配——不是消失，是重心转移。最有价值的能力正在从「写多少代码」转向「做多少 Agent 做不了的判断」。
