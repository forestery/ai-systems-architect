# 附录 F · 12 个月转型计划

> 这份计划原为第 8 章第 8.3 节。单独抽出来作为附录，方便你在不同阶段回来对照，而不需要在宣言式的正文里翻找待办事项。

---

这个计划不是理论设计。是按照我自己和身边人的转型经验画出来的。它假设你现在是一个有 3 年以上经验的软件工程师，每天用 Agent 工具但还没系统地「编排」。如果你经验更少或更多，调整每个阶段的时长。

## 第一阶段：从写代码到用 Agent（第 1-3 个月）

**目标**：停止对比你和 Agent 谁更快。开始把 50% 以上的代码量交给 Agent。

| 周 | 做什么 | 产出 |
|----|--------|------|
| 1-2 | 选一个主力 Agent 工具（Claude Code 或 Codex），所有日常编码都用它。不对比速度——直接让它做。 | 每天至少 50% 代码由 Agent 生成 |
| 3-4 | 不再写单个 prompt 做单个任务。开始写 spec 让 Agent 做完整功能。spec 写得好不好是第一周最重要的判断对象。 | 3 个以上的完整功能 spec |
| 5-8 | 学习 Context Engineering 基础。为你的项目写结构化上下文。尝试分层注入——让 Agent 在不同任务中看到不同深度。 | 第一版结构化上下文文档 |
| 9-12 | 用单 Agent 完成一个完整的、独立的、可以上线的小项目。从 spec 到部署，每一步都是 Agent 做的，你只在关键决策点介入。 | 一个 Agent-driven 的生产项目 |

第一阶段结束时的自检问题：**「Agent 写的代码量超过了我自己写的，而且我没有因此多出 Bug。」** 如果答案是「是」，进第二阶段。如果还不是，找出瓶颈——是 spec 质量不够、上下文不够、还是你不敢放手。

## 第二阶段：编排多 Agent（第 4-6 个月）

**目标**：从一个 Agent 升级到多个 Agent 协作，建立编排直觉。

| 周 | 做什么 | 产出 |
|----|--------|------|
| 13-14 | 学习任务分解。拿一个你已完成的功能，练习反向拆成 3 个 Agent 可执行的子任务。对比你原来的实现和分解后的方案——哪个更干净。 | 5 个以上的任务分解练习 |
| 15-18 | 第一次 Orchestrator-Worker 编排。选一个中等复杂度的功能，用 3 个 Worker 实现，你当 Orchestrator。 | 首次多 Agent 协作项目 |
| 19-22 | 搭建三层验证。不跳过 Layer 2 交叉验证——这是你第一次看到 Agent 的代码被另一个 Agent 挑出毛病。 | 投产的验证流水线 |
| 23-24 | 复盘。过去三个月里，Agent 在什么地方反复出错？你的任务分解有什么模式性的缺陷？写下来。 | 个人编排方法论初稿 |

第二阶段结束时的自检：**「我能把一个模糊的需求拆成 Agent 可以独立执行的子任务，并且在合并前用自动化验证兜底。」**

## 第三阶段：成为架构师（第 7-12 个月）

**目标**：从「会编排」到「能设计编排系统」。你的判断力成为稀缺资产。

| 周 | 做什么 | 产出 |
|----|--------|------|
| 25-28 | 设计 Agent-Native 架构。不再是在现有系统上「加 Agent」——而是让系统本身的模块边界、接口契约、文档结构都是为了让 Agent 高效工作。 | 架构设计文档 |
| 29-32 | 构建生产级 Context Engineering 系统。语义检索、分层注入、上下文刷新——三个组件全搭。跑够第 5 章的三个验收测试。 | 投产的 Context System |
| 33-36 | Agent 安全。攻击面建模、对抗测试、上下文注入攻击防护。 | 安全评估报告 |
| 37-40 | 教别人。把你在前 36 周学到的东西写成内部文档、分享给团队、回答别人的问题。教的过程会暴露你理解不够深的地方。 | 内部培训材料 / 分享记录 |
| 41-48 | 系统化。把你所有的 spec、模板、编排规则、失败案例整理成一套可复用的方法论。这会是你的定价基础。 | 个人方法论库 |
