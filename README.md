## 这本书是写给谁的？

如果你符合以下任意一条，这本书就是为你写的：

- 🟡 **你是一个 3-8 年经验的软件工程师**，看到 Claude Code 和 Codex 在 12 天内写了 44,000 行代码，心里有一个不愿承认的问题：「我的代码能力还剩多少价值？」
- 🟡 **你是一个 Tech Lead 或 EM**，团队从 8 人缩到 4 人，你被要求「学会编排 Agent 而不是管理人」——但没人告诉你具体怎么做
- 🟡 **你是一个 PM 或创业者**，想用 Agent 构建产品，但你需要理解这层技术的边界和架构范式
- 🟢 **你看到了 Gartner 的预测**：到 2026 年，90% 的软件工程师将从「手工编码」转向「AI 流程编排」——你想成为那 10% 的 **AI Systems Architect**，而不是那 90% 中被替代的人

## 这本书在讲什么？

一个核心问题：

> **多 Agent 协同编程，是软件工程的范式革命，还是传统框架内的角色再分工？**

八个递进章节：

```
序  章  一封来自 2026 年的信——为什么你的代码不再重要
        ↓
第 1 章  多 Agent 协同的架构模式（Orchestrator / Planner-Worker-Judge / Gas Town）
        ↓
第 2 章  范式革命还是框架内演进？（SDLC 的阶段还在，但时间尺度变了）
        ↓
第 3 章  角色坍缩：谁留下，谁消失？（初级→架构师的完整迁移路径）
        ↓
第 4 章  Context Engineering：2026 年的承重技能（全新工程学科深度解析）
        ↓
第 5 章  Agent 编排实战（以 Hermes/Codex/Claude Code 为例动手搭建）
        ↓
第 6 章  速度与质量的永恒张力（DORA 数据 + 审查困境 + 安全博弈）
        ↓
第 7 章  成为 AI Systems Architect（能力地图 + 学习路径 + 职业展望）
```
