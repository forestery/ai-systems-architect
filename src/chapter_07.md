# 第 7 章 · 速度与质量的永恒张力

---

2025 年 Google DORA 报告里有一组数据，我每次翻到都会停下来看一会儿。不是因为数据复杂——恰恰相反，它简单得让人不舒服。

| AI 采用增长 | 关联影响 |
|-----------|---------|
| 90% 增长 | Bug 率 **+9%** |
| 90% 增长 | Code Review 时间 **+91%** |
| 90% 增长 | PR 大小 **+154%** |

三行。每一行都在说同一件事：**代码产出在加速，但质量基础设施还停留在原来的速度。**

这不是 AI 的错。Agent 写的代码本身不比人类写的代码 Bug 更多——有些研究甚至显示 Agent 代码在简单任务上的 Bug 率更低。问题是数量。当代码产出速度提升了 10 倍，即使 Bug 率保持不变，绝对 Bug 数量也翻了 10 倍。而你的审查带宽没有翻 10 倍——你的审查带宽还是你一个人。

本章讨论的就是这个张力：**当 Agent 快到让你的质量体系追不上时，你怎么办。** 不是「怎么让 Agent 写得更快」——Agent 已经够快了。

---

## 7.1 DORA 数据说了什么

Google DORA 是 DevOps 领域最受重视的年度报告之一。2025 年的版本第一次把 AI 采用率和质量指标做了关联分析。三个数字值得展开。

**Bug 率 +9%。** 这个数字本身的绝对值不大——但方向很重要。采用 AI 最多的团队，Bug 率不是在下降，是在上升。不是因为 AI 写的代码更差。是因为 AI 写的代码**更多**，而且审查流程没有同步扩展。

**Code Review 时间 +91%。** 这个数字更说明问题。PR 数量在增加，PR 体积在变大（+154%），审查者在面对一个比过去大两倍半的 PR 时，需要花将近两倍的时间来审查。时间翻倍不是因为审查者变慢了——是因为要读的代码翻了一倍多，而人在单位时间内能有效审查的代码行数是一个相对固定的值。

**PR 大小 +154%。** 这是最容易被忽略但最危险的一个趋势。Agent 不擅长「小而频繁的提交」。它倾向于在一个 PR 里塞进整个功能的全部代码——因为它没有「先交一个小版本、等审查通过、再交下一个」的节奏感。这对审查是灾难性的。一个 500 行的 PR，人类可以逐行审查。一个 1500 行的 PR，人类只能「扫一遍，看有没有明显问题」——而 Agent 犯的错，恰恰经常是那种「不仔细看就会漏过去」的错。

Mike Mason 在 2026 年 1 月的那篇文章里有一句总结：“**速度和质量之间的基本张力仍未解决。**” 他没说的是，在 Agent 放大速度之前，这个张力是可管理的——你可以在冲刺结束前加一轮回归测试来兜底。在 Agent 放大速度之后，可管理的窗口消失了。你必须把质量检查嵌入到 Agent 工作的过程中，而不是事后补救。

---

## 7.2 审查瓶颈

当 PR 数量翻倍，PR 体积翻倍，人是无法线性扩展审查带宽的。这是物理约束，不是态度问题。

两个方向可以缓解。

### 方向 1：Agent 审 Agent

人不需要审查每一行代码。人需要审查的是**Agent 审过一遍之后仍然有疑问的东西。**

实现方式是让交叉验证成为流水线的一部分（和 6.4 节的三层验证一致）：

```
Worker 产出 PR
  → Cross-Verify Agent 自动审查
  → 生成审查报告，标记：
      ✅ 安全无问题
      ⚠️ 架构一致性疑问（标记具体位置）
      ❌ 边界条件缺失（指出缺了什么）
  → 人类只看 ⚠️ 和 ❌ 标记的部分
```

我在自己的项目里跑了一个月的交叉验证。数据是这样：

- 100% 的 PR 由 Agent 先审一轮
- 75% 的 PR 在 Agent 审查后没有任何标记——直接合并，人不需要看
- 25% 的 PR 有至少一个标记——人只审查标记部分，平均每个 PR 花 3 分钟

如果没有 Agent 审查，这 100% 的 PR 都需要人看——按每个 PR 平均 15 分钟算，一个月 60 个 PR 就是 15 个小时。Agent 审查把这 15 个小时压到了大约 1 小时。

### 方向 2：控制 PR 大小

Agent 不会自然地「小步提交」。但你可以逼它。

在编排模板里加一条规则：

```yaml
commit_rules:
  max_changed_lines: 300   # 超过 300 行自动拒绝
  max_files: 5             # 超过 5 个文件自动拒绝
  on_violation: "自动拆分为多个 PR"
```

Agent 接到这个规则后，会主动把大改动拆成多个小 PR。每个小 PR 的审查成本低，Agent 审查的准确率也高。

## 7.3 安全：Agent 不该被信任的那一面

Anthropic 的 2026 年报告里有第八条趋势，放在最后但分量不轻：**Agent 同时帮助防御者和攻击者扩展。**

这句话具体是什么意思？

### 防御侧

- 安全扫描可以从「每周跑一次」变成**每次 Agent 提交前都跑一次**
- Agent 可以自动修复常见漏洞——SQL 注入、XSS、不安全的反序列化——修复质量不差
- Agent 可以追踪整个依赖链的 CVE，在发现新漏洞时自动评估项目是否受影响

这些能力在过去需要专门的安全工程师投入大量时间。现在一个编排良好的 Agent 流水线可以覆盖其中的大部分。

### 攻击侧

- 攻击者同样在用 Agent 写漏洞利用代码
- Agent 写得越多，攻击面越大——代码总量在暴增，漏洞检出率不可能同步提升
- 恶意 Agent 可以生成表面上完全正常、但包含隐蔽后门的代码——因为对人类审查者来说，Agent 写的代码本身就有一种「不知道为什么会这样写但看起来没问题」的陌生感，这种陌生感正好掩盖了恶意代码的陌生感

### 一个真实的例子

Yegge 的 Gas Town 数据库下线事件，本质上是安全事件。Agent 被允许操作生产数据库的配置文件，但它不知道这个操作的影响范围。如果当时有一个安全约束——「数据库密码配置 = 不可触碰」——注入到 Agent 的上下文中，这件事可能不会发生。

安全在多 Agent 系统里不是一个「功能」，是一个**上下文约束**。你不能指望 Agent 自己判断什么是危险的——你必须把危险的边界写在它能看到的地方。

## 7.4 信任问题：Agent 能做到，但你不敢让它做

前面讲的是技术层面的速度和质量张力。但还有一个心理层面的，Anthropic 的第四条趋势：**Agent 正在学会主动请求帮助。**

这听起来是个好事。Agent 不确定的时候会问：「这个部分我不太确定，你要不要看一下？」

但这里有一个悖论：

> **Agent 能做的事情越多 → 人类越不确定 Agent 能不能做对 → 人类越不敢委托 → Agent 的价值越无法发挥。**

这不是技术问题——是信任校准问题。人类对 Agent 能力的判断和 Agent 的实际能力之间，存在一个系统性的偏差。偏差的方向取决于你是什么类型的人：

- **过度信任型**：「Agent 跑出来的结果我看都不看就合并了。」——这种人会吃 Gas Town 的亏。
- **过度不信任型**：「Agent 写的每一行我都自己再写一遍确认。」——这种人没有从 Agent 获得任何效率提升，但多了一份审查工作。

两个极端都不是正确答案。正确答案是**渐进式委托。**

### 渐进式委托的三个阶段

**阶段 1：全手动审查。** 你刚开始用 Agent。每个 PR 都逐行审查。持续一个月左右。目的是建立直觉——Agent 在什么类型的任务上稳定？在什么地方容易出错？

**阶段 2：抽样审查。** 你对 Agent 的犯错模式有了大概的判断。低风险 PR（配置文件修改、文档更新、简单重构）合并前只跑自动验证，不再逐行审。高风险 PR（核心模块、安全相关、数据库操作）仍然全审。

**阶段 3：异常驱动的审查。** 你只审查那些自动验证标记了异常的 PR。你的角色从「审查每一行」变成了「审查每一个标记的异常」。前提是你的自动验证流水线已经足够健壮，漏掉致命问题的概率在你可以接受的范围内。

每个阶段的过渡条件很具体：**在阶段 N 的一个月里，自动验证没有漏掉严重问题。** 达到了，升级。没达到，留在当前阶段。信任不是感觉，是记录。

## 7.5 为 Agent 重新设计的质量体系

传统质量体系的建立基于一个前提：**代码是人类写的。** 人类写代码有其固有的犯错模式——边界条件遗漏、并发考虑不足、过于复杂的实现——质量体系围绕这些模式设计。

Agent 写的代码有不同的犯错模式：

- **信息代差**：Agent 基于过时的上下文做决策（Context Engineering 没跟上）
- **统计补齐**：Agent 在信息不足时「编造」合理的补全——不是故意撒谎，是模型的行为特征
- **过度自信**：Agent 从不表达不确定性。即使它的生成概率只有 60%，输出看起来和 90% 一样确定
- **规范偏离**：Agent 可能在 90% 的代码里严格遵守了编码规范，但在剩下 10% 里用了完全不同的风格——因为它处理的是不同批次的任务分解，上下文之间不连续

针对这些模式的质量体系需要三个新组件：

### 组件 1：上下文快照验证

每次 Agent 提交代码时，附带一个「此时 Agent 看到的上下文快照」。审查者（人或交叉验证 Agent）可以对比快照和当前项目状态——如果快照里的 API 契约和当前主分支不一致，这整个 PR 都需要重新做。

### 组件 2：不确定性标记

当前的主要模型不输出不确定性概率。但有一个间接方案：**对同一个任务跑两次，用不同的 Agent 或不同的温度参数。** 如果两次产出高度一致，Agent 对这个任务很有把握。如果两次产出差异明显，这个任务 Agent 不太确定——无论它表象上看起来多自信——人类需要介入。

### 组件 3：历史相似性检索

Agent 提交的代码可能和历史上某次被拒绝的 PR 有相似的结构。在交叉验证阶段，自动检索历史上被拒绝的 PR，如果当前 PR 有相似模式，标记出来让人看。

## 7.6 速度和质量不是二选一

最后这一点很重要。

一个常见的错误结论是：「Agent 太快了，所以我们必须牺牲一部分质量换取速度。」

错。正确的结论是：「Agent 太快了，所以我们必须把质量检查从『人按顺序做』变成『Agent 并行做』。」

质量和速度的矛盾不是 Agent 创造的。这个矛盾在传统软件工程里一直存在——只是在 Agent 放大速度之前，双方是缓慢拉扯。Agent 把速度端的优势拉到了一个让质量端看起来完全追不上的位置。

追上去的方法不是让 Agent 慢下来——是**用 Agent 来做质量检查。** 人类审查带宽是有限的。Agent 审查带宽几乎无限——你每增加一个 Cross-Verify Agent，就增加了一倍的审查能力。质量不再是人的专属责任。它变成了编排架构的一部分。你在设计编排流程的时候，就已经把质量设计进去了——质量检查和工作执行在同一个流水线里并行运转。

这是 Agent-Native 质量体系的核心思想：代码在被写的同一时刻，已经有另一双眼睛在看了。
