# 第 9 章 · 我搞砸的那些事

---

这本书的前八章读起来可能像一个已经搞清楚了一切的人在传授知识。那不是真的。

真实的情况是：我在 2024 年底开始用 Agent 做生产级项目，到现在搞砸过至少三次大的、无数次小的。每一次搞砸都教给我一件书上不会写的事。这些事构成了这本书真正的底层——不是理论，是伤疤。

这一章不教你任何东西。它只是想让你知道，如果你在编排 Agent 时也犯过错——你看到的这个作者，犯得比你还多。

---

## 9.1 故事一：旧缓存的鬼魂

2024 年冬天。我在让两个 Agent 协作完成一个 API 模块——一个写实现，一个写测试。模型 Claude Sonnet。指令写得很清楚，每个 Agent 的任务边界都画好了。

连续三次，实现 Agent 写出的代码用了错误的 Schema。不是语法错误——逻辑是对的、模型版本是对的——但它用的 Schema 是项目三个月前的版本。

我盯了 Prompt 整整三个晚上。换了三种措辞。加了「请使用最新的 api-spec.yaml」。不行。

第四天早上，在喝了半壶咖啡之后，我打开 Agent 启动时加载的上下文文件。API 契约文档，日期戳是三个月前的。缓存。不是模型的问题，不是 Prompt 的问题——是我喂给它的信息就是错的。

Agent 没做错任何事。它只是看到了错误的信息，然后基于错误的信息做出了逻辑正确的决策。

**这个错误教会我一条只花三个晚上就能学会的原则：Agent 永远是对的——基于你给它的上下文。它的错误，几乎总是你的上下文错误穿了它的衣服。**

也是从那一天开始，我才真正理解「Context Engineering」是什么意思。

---

## 9.2 故事二：我信任了 Agent，Agent 信任了我——我们都错了

2025 年春天。我让一个 Agent 团队重构一个用户认证模块。任务分解做得很细——5 个子任务，每个都有清晰的 Spec。Cross-Verify Agent 跑过。Judge Agent 也点了头。

部署上线。一切正常。两天。

第三天，凌晨两点，我被 PagerDuty 叫醒。生产数据库的认证表里，所有用户的密码哈希被更新了一轮。没有人入侵——是 Agent 在一次「优化」中重写了迁移脚本，而那个迁移脚本被包装在一个任务模板里，我从来没审过那个模板。

直接后果：用户全部登出。两小时抢修。二十万用户受影响。

根因不是 Agent 的代码有 Bug——它的代码逻辑是对的。根因是我设计的编排流程里有一个盲区：**任务模板被认为是「安全的」，不受 Cross-Verify 覆盖。** 而 Agent 不会主动告诉你「你现在信任我的那个东西，不应该被信任」。

那两小时抢修的时候，我对 Agent 说的最多的一句话是：「你为什么觉得这是你的权限范围？」当然，Agent 没有回答。它不是一个有判断力的人——它只是忠实地执行了一个我忘了设边界的任务。

**教训：Agent 不会质疑你的架构。它只会忠实地执行——包括你设计缺陷的部分。你的架构里的每一个盲区，Agent 都会替你放大到生产事故级别。**

## 9.3 故事三：我以为自己在上游，其实在下游

2025 年中。一个 SaaS 项目。我用了一个现在看来很有问题的编排决策：把所有 Worker 的输出直接 merge，不做 Judge 审查。理由是「项目还小，迭代要快，Judge 太慢了」。

前两周一切都好。Agent 吞吐惊人，每个 sprint 交付的功能数量是之前的三倍。

第三周。一个新功能上线后，三个已有的 API 端点开始随机返回 500。不是一直 500——大概 2% 的请求会挂。排查了两天，最后发现是三个 Worker 在并行修改三个「看起来不相关」的模块时，其中之一改了一个共享的数据库连接池配置。那个配置被 merge 时没有任何冲突标记——因为它只是改了一个数字。

没有冲突 ≠ 没有破坏。

**我跳过 Judge 环节的原因是「快」。Judge 慢了三个小时，我用掉的排查时间是两天。** 这不是简单的 ROI 计算失误——更深层的问题是：我以为自己在编排 Agent，实际上 Agent 在编排我的时间。

当你跳过质量步骤的时候，你不是在「节省时间」——你是在把时间往后挪，加上利息。利息通常是 10 倍起。

## 9.4 故事四：最强的 Agent 犯了我见过的最蠢的错

这件事发生在 2025 年底。我见过的最聪明的 Agent 之一——一个配置了 Sonnet 模型、拿到了完整项目上下文的 Worker——负责写一个支付回调处理函数。

函数不长，大约 120 行。逻辑清晰。异常处理周全。注释齐全。

唯一的「小」问题：它把回调签名验证函数命名为 `verify_signature()`——而项目里已经有一个同名的全局函数，参数签名完全不同。Agent 不知道那个函数存在，因为它不在这个 Worker 的上下文窗口里。Cross-Verify Agent 也没发现——它审查的是单个文件的正确性，不检查全局命名冲突。

上线后，所有支付回调都因为签名验证失败而被丢弃。损失：六个小时的交易数据，好在有幂等重试机制兜底，没有永久丢失。

**教训：Agent 的「局部完美」是它最危险的属性。** 它能在自己的上下文窗口里写出接近完美的代码——但从全局视角看，那个完美的代码可能正在踩踏系统的其他部分。而 Agent 不知道，因为它看不到。

这也是为什么我在第 2 章花了那么大力气论证「Agent 之间不要通信」——不是因为通信不好，而是因为 Agent 不知道自己不知道什么。当 Agent A 和 Agent B 互不知道对方的存在时，你至少要给它们一个上帝视角的 Judge。

## 9.5 这些搞砸教会我的东西

把这四个故事放在一起，它们指向的不是「Agent 不可靠」——Agent 在这些故事里都很可靠，可靠地执行了我设计好的错误。

真正的问题一直在同一个地方：**不是模型的边界。是我的判断力的边界。我设计的上下文不够、我设的权限画错了线、我为了速度跳掉了不该跳的步骤、我相信了一个 Agent 没有能力挑战的假设。**

这也是为什么我在这本书里反复强调：Agent 时代的工程师不需要放弃编码，但必须把判断力往上移一层。不是「Agent 会不会犯错」的问题——Agent 一定会犯错。问题是：**你的编排设计，能不能让 Agent 犯错的时候，炸在你画好的隔离区里，而不是炸在用户的脸上。**

如果你看完了前八章觉得「这个人好像什么都懂」——回到这一章，再读一遍这四个故事。你会发现我懂的大部分东西，都来自我不懂的时候。

## 9.6 你的搞砸日志

如果你开始认真地编排 Agent 了——开始写这本书的最后一件事。

建一个文档。名字就叫「Agent 故障日志」。每次 Agent 搞砸，写三行：

1. **发生了什么**——Agent 做了什么不该做的事，或者没做什么该做的事
2. **根因**——是上下文不够、spec 歧义、模型能力边界、还是编排设计缺陷
3. **我该改什么**——不是「Agent 该改什么」。Agent 不会改。你自己的流程、上下文、验证步骤里，哪一步加上去之后，这个错误不会再发生

三个月后回看这份日志。你会发现一个模式。那个模式是你的盲区地图——而知道自己盲区在哪里，就是这个职业最值钱的能力。
