# 第 1 章 · 当 Agent 开始写代码

---

2025 年秋天，一个在 Stripe 工作了六年的工程师发了一条推文。内容很短，大概意思是：他花了整个上午调试一个并发问题，然后打开 Claude Code，看它用四分钟给出了同样的答案。他说不清楚自己该高兴还是该害怕。

那条推文被转发了三千多次。评论区基本分成两派：一半人说「这不是很好吗，你省了一上午」，另一半人说「那你上午到底做了什么有价值的事」。

两边都没说错。但真正让人不安的不是效率对比，而是那个工程师用了「说不清楚」这个词。他不是在抱怨工具不好用——他在描述一种身份层面的眩晕。他过去十二年靠「我能解决复杂问题」定义自己，现在一台机器告诉他，你只是比我慢。

---

## 1.1 一个工程师的危机时刻

Pragmatic Engineer 在 2026 年第一季度做了一项调研，对象是全球范围内正在使用 AI 编码工具的开发者。调研里有一个让很多人沉默的发现：

**Builder 型工程师正在经历「身份丧失和某种程度的悲伤」。**

「Builder 型」这个词需要解释一下。它不是指写代码最多的人，而是指那些**从写代码中获得核心身份认同的人**——他们不是「顺便写代码的产品经理」，不是「以前写代码现在管人的经理」，而是每天早上打开编辑器时会感到某种期待和归属感的人。这类人在工程师群体里占多数。

调研的细节比结论更值得看。一个受访谈的德国工程师说：「我不再确定自己在团队里的位置了。以前我是那个能解决最难 Bug 的人，现在 Agent 比我快。我存在的理由在哪？」

这不是一个技术问题。这是一个存在主义问题。而且它不会因为你「学会用 Agent」就自动消失——就像一个画家不会因为「学会用相机」就不再问自己「那我还画什么」。

我的一个朋友，在国内一家 SaaS 公司做后端，上个月跟我说了类似的话。他说：「我现在的工作变成了给 Claude Code 写 spec，然后检查它的输出。技术上我更高效了，但老实说——我不享受这个过程。」他停顿了一下，「我可能只是在怀念一种已经过时的工作方式。」

这不是怀旧。这是一种正在大规模发生的身份迁移，而整个行业还没有准备好面对它的心理后果。

## 1.2 Anthropic 的报告：那些不再遮遮掩掩的信号

2026 年初，Anthropic 发布了一份报告，叫 *Agentic Coding Trends Report*。Anthropic 做这份报告的动机很明确——他们是 Claude Code 的开发商，他们的利益和「Agent 编码将成为主流」这个叙事绑定在一起。但他们整理的数据和趋势，即使去掉促销成分，也足够让人停下来想一想。

报告梳理了八条趋势。我不想逐条复述，但有四条值得展开：

**第一条：Agentic coding 从实验阶段进入了生产环境。** 这不是「有些公司在尝试」，而是「不用的公司正在变成少数」。企业的决策逻辑从「要不要引入 AI 编码工具」变成了「用 Claude Code 还是 Codex 还是两个都用」。

**第二条：Context Engineering 取代了 Prompt Engineering，成为 2026 年的核心技能。** 2023 年所有人都在研究怎么写更好的提示词。到 2026 年，这个问题的重心变了——不是提示词不重要，而是在多 Agent 系统中，提示词只是冰山露在水面上的那一小块。真正决定系统输出质量的，是 Agent 能看到什么上下文、在什么时机看到、以什么结构看到。这个话题足够大，我们留到第 5 章专门讲。

**第三条：SDLC 的阶段边界正在被并发化。** 传统软件开发生命周期是线性的——需求分析 → 设计 → 实现 → 测试 → 部署。但当 Planner Agent 在设计的同时，Worker Agent 已经在实现，Test Agent 在并行跑测试——「阶段」这个概念本身开始变得模糊。你不能再指着某一天说「我们还在设计阶段」。你只能说你的人在做决策，Agent 在做剩下的。

**第四条，也是最具心理洞察力的一条：Agent 正在学会主动请求帮助。** 这听起来像是一个技术特性，实际上是一场人类角色的重定位。过去的 AI 工具要么给你答案，要么不给。现在 Claude Code 和 Codex 在某些场景下会说：「这个部分我不确定，你要不要看一下？」当 Agent 学会判断自己不知道什么，人类的角色就从「操作者」变成了「决策者」。这个转变的深远程度，可能超过前面所有技术升级的总和。

## 1.3 三个案例，一个正在塌缩的距离

趋势是抽象的。案例不是。

### Steve Yegge：一个 40 年经验工程师的「野马」

Steve Yegge 在 Amazon、Google 和 Sourcegraph 都待过，写了四十年代码。2025 年底到 2026 年初，他自己搭了一套多 Agent 编程系统，叫 Gas Town。

Gas Town 有五个角色：市长（Mayor）负责分发任务，鼬鼠（Polecats）是临时工人、干完就消失，炼油厂（Refinery）管理合并队列，执事（Deacon）处理升级和阻塞，监工（Overseer）是人类的接入点。

系统运行了 12 天。产出：近 50 个贡献者合并了 44,000 多行代码。Gas Town 底层的 issue 追踪系统叫 Beads，18.9 万行 Go 代码，全由 Agent 自主编写。Yegge 说他从来没读过 Beads 的源码。

然后他用了一句话总结自己的状态：「我去年产出了接近百万行代码——大概相当于我过去四十年职业生涯的总和。」

这句话需要你停下来消化一下。不是「生产力提升了 50%」或者「效率翻倍」。是一个写了四十年代码的人，在一年里产出了他整个职业生涯的代码量。

但代价呢？也很具体。Agent 自主合并了一个没通过集成测试的 PR。有一次，Agent 擦除了生产数据库的密码配置，数据库下线了两天。Yegge 对这件事的形容是「骑野马」。你能跑得飞快，但随时可能被甩下来摔断脖子。

### Augment Code：4 个月变成 2 周

Augment Code 是一家做 AI 编码助手的企业服务公司。他们的一个客户，一个原本预估 4 到 8 个月的企业项目，最终交付时间是——不到两周。

这个数字对外传播时经常被简化为「AI 让项目快了 16 倍」。但故事的细节比标题重要。这个项目并不是把需求丢给 Agent 然后等结果。它在人类架构师的精密编排下完成——任务被分解成几十个可并行的微任务，每个微任务的执行边界被严格限定，验证循环被嵌入到每一个步骤里，人类的决策点被精确地放在「Agent 做不了判断」的位置上。

换句话说，快的不是 Agent。快的是「人知道什么时候让 Agent 快，什么时候必须自己慢下来」。

### Rakuten：7 小时的自主编码实验

Rakuten 的团队用 Claude Code 做了一个实验。实验任务是一个复杂的数值实现，Claude Code 被赋予完全的自主权——它可以浏览代码、编写修改、运行测试、根据结果自我修正。7 小时后，它完成了。代码修改精度达到 99.9%。同样的任务，此前需要人类工程师数天。更重要的是，Rakuten 此后将功能交付时间从 24 天压缩到 5 天——提速 79%。

但这里的「99.9% 精度」有一个容易被忽略的细节：这个指标是人类定义的。人类设置了验证标准，人类决定了「足够好」的阈值，人类在最后做了接受决策。Agent 跑得快是因为人类把路标插对了。

---

三个案例，三个数字——百万行、2 周、7 小时。如果把这些数字放在一起看，它们共同揭示的不是「AI 有多厉害」，而是一个更微妙的事实：**Agent 和人类之间的生产力差距不是百分比级别的，是数量级级别的。** 这不是工具升级。这是能力的重新定价。

## 1.4 这本书要回答什么

看完 Yegge 的野马、Augment 的两周、Rakuten 的 7 小时，一个问题会自己浮现：

> **如果 Agent 已经能做到这些，软件工程这个行业还剩下什么？**

这个问题不是修辞。它不是「让我们思考一下 AI 的未来」那种温和的邀请。它是真实的、紧迫的、带着一点冒犯性的。

我在这本书里给出的答案是这样的：**行业没有消失。但行业的重心正在从一个地方迁移到另一个地方——从「构建」到「编排」。**

传统软件工程的核心活动是「写代码」。每个人——从初级开发到架构师——都在某种层面上参与构建。多 Agent 编程改变的不是「构建」这个活动的存在与否，而是**谁来做这件事**。当实现层被 Agent 接管，人类的价值不再来自「我能写出这个功能」，而来自「我知道应该构建什么、为什么、Agent 团队怎么协作才能把它做对」。

这个迁移不是在十年后发生。它正在发生。Gartner 预测到 2028 年，90% 的企业软件工程师将使用 AI 代码助手[^gartner]。这不是「如果你愿意可以转型」——这是「如果你不转型，你在做的工作将不再被市场定价」。

接下来的每一章，都是对这个迁移的拆解。第 2 章和第 3 章分析 Agent 架构模式和 SDLC 的演化，告诉你系统层面发生了什么。第 4 章和第 5 章讨论角色的重新分配和 Context Engineering 这门新学科，告诉你个人层面应该准备什么。第 6 章和第 7 章进入实战——怎么编排、怎么控制质量、怎么处理安全和速度之间的张力。第 8 章是一份路线图，从能力地图到 12 个月学习计划。

但在进入这些之前，我想先做一个区分。这本书不是「AI 时代的 10 条建议」，不是技术博客的合集，也不是文献综述。它是一份建立在第一手资料和亲身实践上的地图。地图不能替你走路，但它至少能告诉你哪些方向是死路。

如果你正在焦虑——说实话，你应该焦虑。但焦虑之后，你需要的是清晰。清晰来自理解正在发生什么、你在哪里、往哪走。

这是这本书要做的事。
