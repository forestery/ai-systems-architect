# 第 3 章 · 范式革命还是框架内演进？

---

先说一个故事。

2025 年夏天，一个德国的 Tech Lead 在 Hacker News 上写了一段话。大意是：他们公司的 CTO 在全员会议上宣布，所有新项目从下季度开始，必须使用 AI Agent 完成至少 60% 的代码量。有人举手问：「那我们的 SDLC 流程要不要改？」

CTO 想了三秒钟，说：「不用改。SDLC 还是那六个阶段。只是每个阶段快一点。」

这个 Tech Lead 在帖子里评价：「他说的每个字都对，但整句话全是错的。」

---

SDLC——软件开发生命周期——是软件工程这门学科最基础的框架之一。需求分析 → 设计 → 实现 → 测试 → 部署 → 维护。六个阶段，顺序推进，每个阶段有自己的产出和准入标准。这套框架在过去四十年里承载了几乎所有的软件工程实践。

现在 Agent 来了。问题变得非常具体：这套框架还适用吗？是继续用、小改一下、还是该扔掉了？

本章要做的，是把这个问题的两种回答都展开——不是各打五十大板的综述，而是把双方的强论证摆出来，然后给出一个诚实的判断。

---

## 3.1 六个阶段还在——但每个阶段已经不像自己了

先看一个实际例子。

传统方式开发一个支付模块，周期大概是这样的：

```
Week 1-2:   需求分析 —— PM 写 PRD，和各方对齐
Week 3:     设计 —— 架构师出设计文档，API 契约，数据模型
Week 4-6:   实现 —— 开发写代码，前后端并行
Week 7-8:   测试 —— QA 手动 + 自动测试，Bug 修复循环
Week 9:     部署 —— 灰度发布，监控
Week 10+:   维护 —— 线上问题，迭代需求
```

用多 Agent 编排做同样的模块：

```
上午 9:00   人类描述意图（「实现支付模块，支持微信和支付宝」）
上午 9:15   Planner Agent 生成 spec 和任务分解
上午 9:30   Worker Agent 开始并行执行
               ┌─ Worker A: 后端支付逻辑
               ├─ Worker B: 前端支付界面
               ├─ Worker C: 集成测试
               └─ Worker D: API 文档
下午 2:00   Judge Agent 审查产出，标记 3 个问题
下午 3:00   人类介入审查，确认 2 个问题需要修复
下午 4:30   Worker 修复完成，Judge 通过
下午 5:00   代码合并，部署到 staging
```

十二周变成了一天。但六个阶段都在。

问题在于：「还在」不代表「没变」。一个阶段被压缩到原来的十分之一甚至百分之一时，阶段和阶段之间的关系就变了——不是缩短了，是焊接在一起了。

---

## 3.2 「还在框架内」：这个说法是有道理的

我不是在为保守派辩护。但「SDLC 框架仍然适用」这个说法背后有一些真正有力的论证。忽视它们等于忽视软件工程过去四十年的核心发现。

### 论证 1：阶段结构没有消失，只是被 Agent 化了

2026 年 4 月，一个叫 Kai Xie 的工程师在 Medium 上发了一篇文章，画了一条 AI 驱动的 SDLC 管道：

```
Human Intent → Intent Agent → Product Agent → Design Agent
  → Build Agents (并行执行) → Test Agent → Release Agent
  → Ops Agent → Maintenance Agent
```

我第一反应也是：这不就是把传统 SDLC 的每个阶段前面加上「Agent」吗？

但再想一层，Kai Xie 的观点其实不是「SDLC 还在，所以没变」。他的观点是「SDLC 的阶段结构是对的，变的是执行者」——人在做需求分析的阶段需求和 Agent 在做是一样的，但人在做和 Agent 在做的速度、覆盖面和犯错模式完全不同。

### 论证 2：角色映射非常清晰

如果你把多 Agent 系统中的角色和传统软件工程团队的角色放在一起看，映射关系几乎是 1:1 的：

| Agent 角色 | 传统角色 | 关键区别 |
|-----------|---------|---------|
| Orchestrator | Tech Lead | 编的是 Agent，不是人 |
| Planner | PM / 架构师 | 产出 spec，不是 PRD |
| Worker | 开发者 | Agent 替人写代码 |
| Judge | QA | 全覆盖，不是抽样 |
| Deacon / Overseer | 人类监督者 | 异常介入，不微观管理 |

如果你可以画出这张映射表——角色没多没少，只是执行者变了——那你说「软件工程的框架变了」，说服力不够。

### 论证 3：经典 SE 原则在 Agent 层变得更加重要

Mike Mason 是 ThoughtWorks 的技术主管。他在 2026 年 1 月的一篇文章里写了一段我很认同的话：

> 「层级化架构、Git 记忆系统、上下文工程、验证循环——控制多 Agent 输出混乱所需的技术机制，本身就是经典软件工程的解决方案。」

模块化。关注点分离。接口契约。测试驱动开发。这些原则在传统 SE 中重要，在多 Agent 编排中**更加**重要——因为 Agent 不会主动遵守它们，你必须把它们刻进编排架构里。

好的 SE 实践不是被 Agent 废掉了。是被 Agent 逼着升级了。

---

## 3.3 「正在脱离框架」：这些论证也不容忽视

以上是「还在框架内」这一侧。但另一侧的论证同样有力，而且指向一些更深的变化。

### 论证 1：时间尺度发生了质变

Augment Code 把 4-8 个月的项目压缩到 2 周。Rakuten 把数天的工作压缩到 7 小时。这不是「快了 50%」——是**数量级的变化**。

当一个阶段的时间短于人类认知的切换时间，「阶段」这个概念本身的意义就变了。传统 SDLC 中，「需求分析阶段」的存在前提是「需求分析需要足够长的时间，长到你可以把它单独命名、单独排期、单独配人」。当需求分析只需要 30 分钟，它不是「一个阶段」——它是「Planner Agent 启动时做的第一件事」。

阶段还在，但阶段的**独立存在意义**消失了。

### 论证 2：从顺序走向并发——这不是小改

传统 SDLC 是严格顺序的。设计完成才能实现。实现完成才能测试。每个阶段的产出是下一个阶段的准入条件。

在多 Agent 模式下：

```
传统 SDLC 的时间线：
  需求 ──→ 设计 ──→ 实现 ──→ 测试 ──→ 部署
  (串行，一个完成再开始下一个)

Agentic SDLC 的时间线：
  需求 ──→ 设计 ──→ 实现 ──→ 部署
            └──→ 测试 ──┘
            └──→ 文档 ──┘
  (并发，多个阶段同时进行)
```

并发不只是「更快」。并发意味着**阶段之间的反馈回路被打通了。** 测试 Agent 在实现 Agent 还在写代码的时候就开始跑测试。测试结果实时反馈给实现 Agent，实现 Agent 在同一个执行周期内修复问题。

这在传统 SDLC 里是不可能的——因为传统 SDLC 的阶段边界是人定义的，而人的注意力一次只能处理一个阶段。Agent 没有这个限制。它可以同时「在设计」「在实现」「在测试」——因为这三个活动不是三个 Agent 的独占域，而是三个并行线程。

### 论证 3：人类价值从「构建」迁移到了「编排」——这不是渐进改良

Forbes 技术委员会在 2026 年 3 月有一篇标题就很直白的文章：*How AI Broke The Software Development Life Cycle*。

文章的核心论点：传统 SDLC 是为人类的线性认知模式设计的。当 AI 打破了这种线性——同时设计、同时实现、同时测试——SDLC 框架就从一个「流程」变成了一个「参考坐标系」。它还在，但你不再按它的步骤走。

更大的变化在人的一端。人类从「构建」迁移到「编排」不是技能升级——是职业身份的重定义。代码从「我的作品」变成了「Agent 输出的产物，我来审查」。这个迁移影响的不只是 SDLC 的某个阶段，而是**人参与 SDLC 的方式本身。**

---

## 3.4 一个诚实的判断

我不会说「两边的论证都成立」。我会说：**两边的论证指向的是同一件事的不同层次。**

在**结构层面**，SDLC 框架仍然成立。六个阶段的逻辑没有错——软件工程还是需要理解需求、设计方案、产出代码、验证质量、交付运行、持续维护。任何一个阶段跳过不做，系统就会在某个地方出问题。

在**运行层面**，SDLC 的运行方式发生了框架内部的重构。阶段之间的边界被并发化。时间尺度被压缩 10 到 100 倍。人类从执行层退到审查层和决策层。Agent 接管了实现和测试，人类的精力从「怎么做」转移到了「做什么」和「为什么」。

这不是库恩意义上的「范式革命」——旧范式没有被抛弃，SDLC 的分类逻辑仍然有用。但也不是「渐进改良」——角色的重新定位和速度的量级变化加在一起，已经超出了「改良」这个词能承载的范围。

我找到的最接近的比喻是飞机。

> **飞机没有脱离物理定律。它没有让重力消失。但它改变了我们对「旅行」的理解——从「几天」变成了「几小时」，从「冒险」变成了「通勤」，从少数人的体验变成了大众的基础设施。**

多 Agent 编程也是这样。**它没有脱离软件工程。它没有让 SDLC 消失。但它正在改变我们对「构建软件」的理解——从「人写代码」变成了「人编排 Agent」，从「周和月」变成了「小时和天」，从「手工作坊」变成了「流水线」。**

这个比喻的关键不在「飞机很厉害」——而在「物理定律没变」。软件工程的原则没变。变的只是执行的规模、速度和分配方式。而这三个变化加在一起，分量已经和「范式革命」差不多了。

---

## 3.5 这对你意味着什么

如果你是一个还在每天写代码的工程师，这一章的结论可能会让你不舒服。不是因为结论激进——恰恰相反，结论很保守。它说的是：**你没做错什么，但你做的事情的价值正在被重新定价。**

代码从「稀缺技能」变成「Agent 可以无限产生的产物」——这个转变已经开始。SDLC 不会被废除，但你在 SDLC 里参与的那个环节——实现——正在被 Agent 覆盖。

接下来的两章会回答「那你该去哪」——第 4 章讲角色的重新分配，第 5 章讲你该建立什么新能力。

但在进入下一步之前，先把这一章的核心判断记住：**这场转变不是要推翻软件工程，而是让你从一个位置移到另一个位置。不移的人不会被淘汰——他们只是会发现自己做的工作，不再被市场按从前的价格支付。**
